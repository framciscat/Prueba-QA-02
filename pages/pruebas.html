Tipos de prueba durante el proceso de desarrollo y ejecución de Software

Si existe, es posible probarlo

Pruebas, pruebas y más pruebas

Como Analista QA, nuestra labor va a radicar en realizar diferentes pruebas para diferentes aspectos que se desean evaluar, todo con el fin de garantizar la calidad del producto y que este cumpla con los requerimientos del cliente y necesidades y expectativas del usuario. Sin embargo, ¿qué tipos de pruebas podemos encontrar a lo largo de todo el proceso de pruebas? Entérate en el siguiente artículo.

Niveles de prueba: De acuerdo a la etapa del ciclo de desarrollo de software en que se encuentre el proyecto, las pruebas se dividirán de forma general, en niveles. Estos niveles comprenden, pruebas de componentes, pruebas de integración, pruebas de sistema y pruebas de aceptación.

Pruebas de componentes: Primera fase de pruebas dinámicas, se realizan sobre cada módulo o componente que se pueda probar de forma independiente. También conocido como prueba unitaria o de módulo.Un módulo, corresponde a una unidad mínima funcional del sistema. Las pruebas dinámicas corresponden al momento en que se ejecuta el software y permite comprobar su correcta codificación, tratando errores de manera más controlada.

Pruebas de integración: Estas pruebas se encargan de asegurar la comunicación, enlaces y datos compartidos entre módulos o diferentes partes del sistema, garantizando que funcionen correctamente. Antes de las pruebas de integración, las pruebas unitarias deben haber pasado.Pueden cubrir integración de interfaces provistas por organizaciones externas. Ejemplo: Servicios web.

Pruebas de sistema: Verifican que se han integrado adecuadamente los elementos del sistema y que funcionan adecuadamente como un todo. Similar a las de integración pero con un alcance más amplio. Prueba final a nivel de desarrollo, para verificar que el sistema cumple con las especificaciones.

Pruebas de aceptación: Luego de haber realizado las pruebas de sistema y haber corregido la mayoría de defectos. El sistema será entregado a los usuarios y otros interesados para realizar pruebas de aceptación. El entorno de pruebas debe ser lo más parecido al entorno de producción.

Tipos de pruebas de Software: Grupo de actividades que se centra en un objetivo particular. Evaluar características funcionales, evaluar características no funcionales, evaluar estructura/ arquitectura. Evaluar los efectos de los cambios.

Pruebas funcionales Responden a lo que debería hacer el sistema, obteniendo una cobertura funcional, que radica en cuantos componentes del sistema han sido probados. Algunos ejemplos de pruebas funcionales, serían las pruebas de caja negra y caja blanca.

Pruebas no funcionales Conocido como prueba de las características del producto de software. Se evalúa “qué tan bien o qué tan rápido” se realiza algo. Ejemplo: tiempo de respuesta del sistema. Se realizan en todos los niveles de prueba y deben ejecutarse tan pronto como sea posible. Si se descubren defectos no funcionales, pueden ser altamente riesgosos para el éxito de un proyecto.

Pruebas no funcionales que encontraremos:

Pruebas de recuperación: Verificar qué tan rápido o que tan bien se recupera una aplicación luego de experimentar un fallo de software o hardware.

Prueba de mantenibilidad: Evaluar qué tan fácil es realizar el mantenimiento de un sistema o aplicación.

Pruebas de seguridad: Probar características de seguridad del sistema. Si el sistema es seguro o no, si puede ser vulnerado, si existe control de acceso por medio de cuentas de usuario.

Pruebas de resistencia: Evalúa someter al sistema a una carga determinada durante un periodo de tiempo para determinar cómo se comporta durante un uso prolongado

Pruebas de volumen: Validar el funcionamiento de la aplicación con ciertos volúmenes de datos.

Pruebas de estrés: Pruebas de carga que se realizan con demandas mayores a la capacidad operativa. Con frecuencia hasta llegar al tiempo de ruptura.

Pruebas de carga: simular demandas sobre una aplicación de software y medir el resultado. Se evalúan en condiciones de sobrecarga o picos en la demanda.

Cobertura no funcional: Es la medida de que algún elemento no funcional ha sido ejercitado mediante pruebas. Se expresa en % de elementos cubiertos.

Pruebas de caja blanca: Se derivan pruebas basadas en la estructura o en la implementación interna del sistema. Nos interesa ver lo que sucede dentro de la caja: Es decir, código, arquitectura o flujos de datos dentro del sistema.

Dentro de las técnicas que encontraremos en caja blanca están:

Prueba y cobertura de sentencia: Consta en verificar que cada línea de código haya sido visitada o ejecutada al menos una vez, lo que ayuda a identificar partes del código que no se han probado. Prueba de cobertura y decisión: Evaluar si todas las decisiones en el código fuente han sido probadas. Una decisión es una estructura de control, como una instrucción "if", que determina la dirección de ejecución del programa.



Pruebas relacionadas a cambio: Permite verificar que los cambios han corregido el defecto o la funcionalidad se ha implementado correctamente y no genera un efecto negativo.

Se pueden encontrar las siguientes pruebas relacionadas al cambio:

Pruebas de confirmación: Confirma si el defecto detectado fue reparado. Luego debe ser probado con los casos de prueba que fallaron. Pruebas de regresión: Cambio realizado en el código o corrección puede afectar accidentalmente el comportamiento de otras partes (mismo componente, otros componentes o incluso en otro sistema). Verificar que la reparación de un defecto no haya causado otros defectos.

Pruebas de mantenimiento: Evaluar el éxito de los cambios (reparaciones, nuevas funciones, eliminar o alterar funciones) o verificar efectos secundarios.



Pruebas de caja negra: Aquellas pruebas en que no se requiere saber cómo funciona el software, sino los resultados que se obtienen al probarlo.

Algunas pruebas de caja negra que son posibles encontrar son las siguientes:

Partición de equivalencia: Consta en dividir un conjunto de condiciones de prueba en grupos que se pueden considerar iguales. Ejemplo: probar entre votantes mayores y menores de edad. Se encuentran particiones válidas e inválidas.

Análisis de valores frontera o valores límite: Extensión de la partición de equivalencia, sólo se utiliza cuando la partición está ordenada en datos numéricos o secuenciales. Mínimo o máximo son sus valores frontera.

Pruebas de tabla de decisión: proporcionan una forma sistemática de reglas de negocio compleja. Ayudan a la selección eficiente de casos de prueba. Pasos: identificar entradas o condiciones, identificar acciones o salidas e identificar combinaciones.

Prueba transición de estado: Se utiliza cuando el proceso o sistema bajo análisis encaja con máquina de estados finitos. El sistema se maneja en un número limitado de estados.Se muestran posibles estados de software, entrada, salida y realiza las transiciones entre estados.

Prueba de caso de uso: Descripción de un uso particular de un sistema. Cada caso de uso describe las interacciones del actor con el sistema para lograr una tarea específica. Se describe lo que hace y ve el actor, no el sistema. Se utilizan en niveles de prueba de sistema y aceptación.

Pruebas basadas en la experiencia: Se basa en el conocimiento de los participantes para aplicar las pruebas.

Algunas pruebas basadas en la experiencia que encontramos son las siguientes :

Predicción de errores: Anticipar la ocurrencia de defectos, fallas, basándose en la experiencia del probador.

Pruebas exploratorias: Explorar y conocer el software. Podemos responder preguntas como ¿Qué hace el software?, ¿Qué no hace el software?.

Pruebas basadas en listas de comprobación: Es un conjunto de preguntas basadas en defectos potenciales, por qué y cómo falla el software. Esta lista puede ser corregida o ampliada. Proporcionar directrices cuando no hay casos de prueba.



Pruebas de aceptación: Como mencionamos con anterioridad, las pruebas de aceptación se realizan luego de haber corregido la mayoría de defectos para su entrega a los usuarios y partes interesadas.

Algunas pruebas que es posible encontrar:

Pruebas de aceptación de usuario UAT: Se centra en validar que el sistema cumple los requisitos de funcionamiento esperado en un entorno operativo real o simulado. Su objetivo es construir confianza, cumplir requisitos y realizar procesos de negocio.

Pruebas de aceptación operacional OAT o pruebas de aceptación de producción: Se valida que el sistema cumple los requisitos de operación. Objetivo: construir confianza en que el sistema funcionará correctamente para los usuarios en el entorno operativo.

Pruebas de aceptación contractuales y regulatorias: Se realizan para cualquier reglamentación y normativa que deba cumplirse: gubernamentales, legales o de seguridad. Objetivo: lograr cumplimiento contractual o reglamentado.

Pruebas de aceptación Alfa y Beta: Suelen ser utilizados por desarrolladores de un producto comercial que desean recibir feedback antes de liberar el producto al mercado. Objetivo: Generar confianza entre clientes potenciales.

Un plan de pruebas exhaustivo y pensado en las necesidades del usuario y requisitos del cliente, será crucial para que como probadores, podamos garantizar la calidad del software.
